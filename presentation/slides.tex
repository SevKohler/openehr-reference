\section{What is a clinical reference?}
% ─────────────────────────────────────────────────────────────────────────────

\begin{frame}{What is a clinical reference?}
  Clinical data rarely stands alone. A procedure has a reason. A finding has a site.
  A device belongs to an entry. A composition belongs to an encounter.

  \bigskip
  A \textbf{clinical reference} is a typed pointer from one piece of clinical data
  to another — inside the same record, or to an external system.

  \bigskip
  \begin{alertblock}{The question}
    How do you model that pointer so it is typed, constrainable, and interoperable?
  \end{alertblock}
\end{frame}

% ─────────────────────────────────────────────────────────────────────────────
\section{How FHIR solves it}
% ─────────────────────────────────────────────────────────────────────────────

\begin{frame}{FHIR: \texttt{Reference}}
  FHIR models references as a first-class data type, used \textbf{explicitly}
  wherever a resource field points to another resource.

  \bigskip
  \texttt{Reference} carries: a literal reference string, a type, a logical
  identifier, and a human-readable display label.

  \begin{figure}
    \includegraphics[width=0.75\textwidth]{../Reference.png}
  \end{figure}
\end{frame}

\begin{frame}{FHIR: \texttt{CodeableReference}}
  \texttt{CodeableReference} goes one step further: the field may hold
  \textbf{either} a coded concept \textbf{or} a \texttt{Reference} — or both.

  \bigskip
  Example: \texttt{Procedure.reason} can be a free code when nothing is linked,
  or a reference to a \texttt{Condition} resource when one exists.

  \begin{figure}
    \includegraphics[width=0.6\textwidth]{../CodeableReference.png}
  \end{figure}
\end{frame}

\begin{frame}{What FHIR gets right}
  \begin{itemize}
    \item References are \textbf{explicitly typed} — not free text, not implicit
    \item The pointer and a coded fallback live \textbf{in the same field}
    \item References are \textbf{constrainable} per resource field — not attachable everywhere
    \item Interoperability is enforced because the structure is fixed
  \end{itemize}
\end{frame}

% ─────────────────────────────────────────────────────────────────────────────
\section{How openEHR handles it today}
% ─────────────────────────────────────────────────────────────────────────────

\begin{frame}{openEHR: \texttt{LINK}}
  \texttt{LINK} is available on \textbf{every} \texttt{LOCATABLE}. It carries a
  meaning, a type (free string), and a target URI.

  \begin{figure}
    \includegraphics[width=0.72\textwidth]{../LINKS.png}
  \end{figure}
\end{frame}

\begin{frame}{openEHR: external references}
  External references are modeled separately in the RM via \texttt{PARTY\_REF},
  \texttt{OBJECT\_REF}, and related classes — a different mechanism from \texttt{LINK}.

  \begin{figure}
    \includegraphics[width=0.72\textwidth]{../PARTY.png}
  \end{figure}
\end{frame}

% ─────────────────────────────────────────────────────────────────────────────
\section{The problems}
% ─────────────────────────────────────────────────────────────────────────────

\begin{frame}{Problem 1: \texttt{LINK} is too permissive}
  \begin{itemize}
    \item \texttt{LINK} can be attached to \textbf{any} \texttt{LOCATABLE} — there
          is no way to restrict where it appears
    \item \texttt{type} is a \textbf{free string} — no enforcement, no terminology
          binding, no interoperability
    \item In practice every team uses it differently, making cross-system querying
          unreliable
  \end{itemize}

  \bigskip
  \begin{alertblock}{}
    The same chaos that motivated typed references in FHIR exists in openEHR today,
    just dressed differently.
  \end{alertblock}
\end{frame}

\begin{frame}{Problem 2: no coded fallback}
  FHIR's \texttt{CodeableReference} lets a field hold a code \textbf{or} a reference.

  \bigskip
  In openEHR there is no equivalent. A field is either a \texttt{DV\_CODED\_TEXT}
  or it carries a \texttt{LINK} on the side — two unrelated mechanisms with no
  shared semantics and no archetype-level constraint tying them together.

  \bigskip
  Example: \texttt{Procedure.reason} should accept a static code when no linked
  diagnosis exists, and a typed reference when one does. Today this cannot be
  expressed cleanly in one field.
\end{frame}

\begin{frame}{Problem 3: two separate mechanisms, no bridge}
  \begin{columns}[T]
    \column{0.48\textwidth}
      \textbf{Internal (\texttt{LINK})}
      \begin{itemize}
        \item Points inside the EHR
        \item On every \texttt{LOCATABLE}
        \item Free-string type
      \end{itemize}

    \column{0.48\textwidth}
      \textbf{External (\texttt{OBJECT\_REF} etc.)}
      \begin{itemize}
        \item Hard-coded in the RM for specific classes (PARTY etc.)
        \item Cannot be used inside \texttt{ELEMENT.value}
        \item No coded fallback
      \end{itemize}
  \end{columns}

  \bigskip
  There is no single mechanism that covers both internal and external targets,
  is usable inside an archetype field, and supports a coded fallback.
\end{frame}

\begin{frame}{Problem 4: no value proxying}
  FHIR lets a reference surface data from its target inline — the \texttt{display}
  field carries a human-readable label resolved from the linked resource.

  \bigskip
  In openEHR, if a procedure needs to show the diagnosis name from the linked
  \texttt{problem\_diagnosis}, the only option today is to \textbf{duplicate} the
  value. There is no mechanism to say ``resolve this value from the target at
  recording time and store it here.''

  \bigskip
  Duplication breaks when the source is updated and the copy is not.
\end{frame}

\begin{frame}{Summary: what is missing}
  \begin{tabular}{lcc}
    \toprule
    Capability & FHIR & openEHR today \\
    \midrule
    Typed, constrainable reference     & \checkmark & \texttimes \\
    Usable inside an archetype field   & \checkmark & \texttimes \\
    Internal + external in one type    & \checkmark & \texttimes \\
    Coded fallback in same field       & \checkmark & \texttimes \\
    Value proxying from target         & (display)  & \texttimes \\
    \bottomrule
  \end{tabular}
\end{frame}
